# tianfu_net_gov - 分类引擎和风险预警优化建议

## 分类引擎（CybClassificationServiceImpl）

### 现有逻辑
纯规则引擎：行业类型(30%) + 注册类型(25%) + 经营状态(20%) + 资本类型(15%) + 规模(10%)，硬编码分数。

### 存在的问题

**1. 分数权重写死在代码里，无法动态调整**
- 监管部门调整权重需要改代码 → 重新部署

**2. 没有权重校验逻辑**
- 5个维度加总必须=100%，代码里没有断言
- 有人改配置漏了，总分就不是100%

**3. 分类结果没有"置信度"概念**
- 企业同时满足多个分类条件时，没有优先级机制
- 结果只有 A/B/C 三档，太粗糙

**4. `batchClassify()` 是一次性全量处理**
- 没有增量处理（只处理新增/变更的企业）
- 每次全量跑，数据量大了会很慢

### 建议

| 建议 | 说明 |
|------|------|
| 权重表独立出来 | 新建 `cyb_classification_weight_config` 表，后台可配置 |
| 增量分类 | 加时间戳字段，只处理 `updated_at > last_run_at` 的企业 |
| 分类过程可审计 | 每次分类记录详细的打分明细，方便追溯 |
| 阈值动态化 | A/B/C 分类的分数线也做成配置，而不是固定 80/60 |

## 风险预警（CybRiskAlertServiceImpl）

### 现有逻辑
目前是**人工触发**扫描 + 手动标记处理。`triggerEnterpriseRiskScan` 是空实现。

### 存在的问题

**1. 预警完全依赖人工触发**
- 没有自动扫描机制

**2. 没有"定时自动扫描"机制**
- 没有内置的定时任务调度

**3. 预警规则没有独立配置**
- 红/橙/黄/蓝 四色预警规则写死在 SQL 里

**4. 预警没有"升级机制"**
- 已处理的问题如果没有改善，是否需要升级？没有

### 建议

| 建议 | 说明 |
|------|------|
| 实现自动定时扫描 | 用 `@Scheduled` 或 Quartz，每小时/每天自动跑一次 |
| 预警规则独立配置表 | 新建 `cyb_risk_alert_rule` 表，定义每种预警类型的触发条件 |
| 预警升级机制 | 超时未处理自动升级（红→橙→黄） |
| 预警推送渠道 | 目前只有数据库记录，建议加 Email/钉钉/微信通知 |
| 预警统计分析 | `getAlertStatistics()` 需落地——统计各等级预警数量、处理率、平均处理时长 |

## 两个模块的联动问题

当前分类和预警**完全独立**，没有联动：
- 分类结果（A/B/C级）应该影响预警的敏感度（A级企业应该查得更严）
- 但代码里没有这个关联

**建议：** 在预警规则里加一个字段：`enterprise_level`，高分企业（分类A）用更严格的预警阈值。

## 总结优先级

| 优先级 | 改动 | 难度 |
|--------|------|------|
| 🔴 高 | 预警统计 `getAlertStatistics()` 落地 | 低 |
| 🔴 高 | 定时自动扫描机制 | 中 |
| 🟡 中 | 分类权重配置化 | 中 |
| 🟡 中 | 预警规则配置化 | 中 |
| 🟢 低 | 分类-预警联动 | 高（涉及模型改动）|
